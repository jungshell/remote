# 목표–하위 업무 관리 방안

**최종 목표(한 개)**에 대해, 그걸 진행하면서 **파생되는 하위 업무들**을 묶어서 효율적으로 관리하는 방안입니다.

---

## 1. 데이터 구조

### 개념
- **목표(Goal)** = 최종 목표 하나. 예: "Q1 마케팅 캠페인 완료".
- **하위 업무(Sub-task)** = 그 목표를 이루기 위해 파생된 작업. 예: "기획서 작성", "디자인 시안 확정", "배포" 등.

### 구현 방식 (선택지)

| 방식 | 설명 | 장점 | 단점 |
|------|------|------|------|
| **A. 자기 참조** | `Task`에 `parentTaskId`(선택) 추가. 없으면 최상위(목표 또는 단독 할 일), 있으면 해당 목표의 하위 업무. | 기존 할 일 구조 그대로 활용, 한 컬렉션으로 조회 가능. | "목표 vs 일반 할 일" 구분을 별도 필드로 할지 결정 필요. |
| **B. 목표 전용 컬렉션** | `goals` 컬렉션 + `tasks`에 `goalId`로 연결. | 목표와 일반 할 일이 명확히 분리. | 목표/할 일 두 종류 관리, 조회 시 조인 또는 두 번 읽기. |

**권장: A. 자기 참조**  
- `Task`에 `parentTaskId?: string` (선택) 추가.  
- `parentTaskId`가 **없는** 할 일 = 최상위. 이 중 "목표"로 쓰고 싶은 것은 **`isGoal?: boolean`** 또는 **목표 전용 보기**에서만 "목표"로 취급해도 됨.  
- `parentTaskId`가 **있는** 할 일 = 해당 ID를 가진 할 일의 하위 업무.

### Firestore 구조 예시 (A안)

```
tasks (기존)
├── id: "goal1"
│   title: "Q1 마케팅 캠페인 완료"
│   parentTaskId: (없음)     ← 최상위 = 목표
│   isGoal: true             ← 선택: 목표 여부
│   ...
├── id: "task1"
│   title: "기획서 작성"
│   parentTaskId: "goal1"     ← goal1의 하위
│   ...
├── id: "task2"
│   title: "디자인 시안 확정"
│   parentTaskId: "goal1"
│   ...
```

- **목표만 보기**: `parentTaskId == null` (또는 `isGoal == true`) 필터.  
- **특정 목표의 하위만 보기**: `parentTaskId == goalId`.  
- **진행률**: 해당 목표의 하위 중 `completed === true` 개수 / 전체 하위 개수.

## 2. UI/UX 제안

### 목록 화면
- **보기 전환**: "전체" / "목표만" / "단독 할 일만" (하위 없는 최상위).
- **목표 카드**  
  - 제목, 마감일, **진행률(예: 2/5 완료)** 프로그레스 바.  
  - 클릭 또는 "펼치기" 시 **하위 업무 목록** 표시 (들여쓰기 또는 트리).  
  - 하위에 "추가" 버튼으로 바로 하위 업무 생성.

### 목표 상세(또는 펼침 영역)
- 목표 1개 + 그 하위 목록을 한 화면에.
- 하위 업무: 체크박스(완료), 제목, 마감일, 수정/삭제.
- **"하위 업무 추가"** 입력창: 제목 입력 후 추가 시 `parentTaskId = 현재 목표 ID`로 저장.
- **진행률 자동 반영**: 하위 완료 시 퍼센트 또는 "n/m 완료" 갱신.

### 추가 플로우
- **새 할 일 추가 시** "단독 할 일" / "목표로 추가" / "○○ 목표의 하위로 추가" 중 선택 (드롭다운 또는 모달).
- **기존 할 일을 목표로 승격**: 수정에서 "목표로 설정" 시 `isGoal = true`, 기존에는 `parentTaskId` 없음 유지.
- **기존 할 일을 특정 목표의 하위로**: 수정에서 "상위 목표 선택" 드롭다운에서 목표 선택 → `parentTaskId` 설정.

### 주간/월간 뷰
- 목표는 **한 줄 요약**(제목 + 진행률)만 표시하고, 클릭 시 하위를 펼치거나 상세로 이동.
- 하위 업무는 마감일 기준으로 기존처럼 요일/달에 표시.

## 3. 효율적 관리 포인트

| 항목 | 내용 |
|------|------|
| **한 화면에서 목표 단위로 보기** | 목표 카드 펼치면 하위만 보이게 해서, "이 목표 때문에 뭘 해야 하지?"를 바로 파악. |
| **진행률 자동 계산** | 하위 완료 시 자동으로 n/m, % 표시. 목표 카드·상세 모두 동일 숫자. |
| **빠른 하위 추가** | 목표 카드 또는 상세에서 "하위 추가" 한 줄 입력으로 `parentTaskId` 자동 설정. |
| **필터/정렬** | "이 목표의 할 일만" 필터, 정렬은 마감일/우선순위 등 기존과 동일. |
| **목표별 색/아이콘(선택)** | 목표마다 라벨 색을 두면 시각적으로 구분 가능. |

---

## 4. 구현 시 필요한 작업

1. **타입·Firestore**  
   - `Task`에 `parentTaskId?: string`, `isGoal?: boolean` 추가.  
   - 기존 할 일은 둘 다 없음 → "단독 할 일"로 동작.

2. **API·쿼리**  
   - 목표만: `parentTaskId == null` (및 선택 시 `isGoal == true`).  
   - 하위 목록: `parentTaskId == goalId`.  
   - Firestore 인덱스: `ownerId` + `parentTaskId` (및 필요 시 `createdAt` 등).

3. **화면**  
   - 목록: 목표 카드 + 펼침 시 하위 리스트, 진행률 바, "하위 추가".  
   - 추가/수정: "상위 목표" 선택 또는 "목표로 추가" 옵션.

4. **보안 규칙**  
   - `parentTaskId`가 있으면, 참조하는 상위 할 일이 같은 `ownerId`인지 검증(선택). 또는 기존처럼 `ownerId`만 검사.

이 방안으로 구현하면 "최종 목표 하나 → 거기서 파생된 업무들"을 한 덩어리로 관리할 수 있습니다. 원하시면 `Task` 스키마 변경부터 구체 코드안으로 이어서 정리하겠습니다.
