# 미구현 킵 항목 — 단계별 구현 방향

아래 순서대로 진행하면 의존성·난이도를 고려한 구현이 가능합니다.

---

## 포트 설정 (적용 완료)

- **개발**: `npm run dev` → **3005**
- **프로덕션**: `npm run start` → **3005**
- 다른 앱과 포트 중복을 피하려면 `.env.local`에 `NEXT_PUBLIC_BASE_URL=http://localhost:3005` 설정 권장.

---

## 1단계: 일기 수정/편집 (3번) ✅ 완료

**목표**: 기존 일기 내용을 수정해 다시 저장.

| 구분 | 내용 |
|------|------|
| API | `PATCH /api/diary` — `id` + 수정 가능 필드(summary, quote, timeline, goodThingsByMember, keywords, location, weather 등). Firestore `doc(id).update(partial)` |
| UI | 풀스크린 읽기 화면에 **「수정」** 버튼 → 편집 모달/화면(텍스트 필드들) → 저장 시 PATCH 호출 후 리스트/읽기 뷰 갱신 |
| 상태 | 수정 중인 일기 id, 낙관적 업데이트 또는 저장 후 다시 fetch |

**진행 순서**: API PATCH 구현 → ReadFullScreen에 수정 버튼 연결 → 편집 폼 모달 컴포넌트 → 저장 후 토스트 + 닫기.

---

## 2단계: 오늘과 비슷한 날 추천 (2번) ✅ 완료

**목표**: 현재 보고 있는 일기와 비슷한 과거 일기를 추천.

| 구분 | 내용 |
|------|------|
| API | `GET /api/diary?similarTo=<id>` 또는 `GET /api/similar?id=<id>`. 해당 일기의 keywords/date/location 활용, 같은 키워드·비슷한 날짜·같은 장소 가중치로 유사도 계산 후 상위 3~5건 반환 |
| UI | ReadFullScreen 하단에 **「이런 날도 있어요」** 섹션 → 카드 3~5개(날짜, 한 줄 요약) → 클릭 시 해당 일기 풀스크린으로 표시 |
| 데이터 | 기존 일기 목록(또는 API)에서 키워드/날짜/장소만 사용. LLM 없이 구현 가능. |

**진행 순서**: similar 로직(키워드 겹침 개수 + 날짜 차이) → API 라우트 → ReadFullScreen에서 similarTo fetch 후 블록 렌더.

---

## 3단계: 사진 앨범 뷰 (4번) ✅ 완료

**목표**: 일기와 무관하게 **모든 업로드 사진**을 그리드로 보기.

| 구분 | 내용 |
|------|------|
| API | `GET /api/diary?limit=100` 등으로 목록 확보 후 클라이언트에서 `photoUrls` 플래트닝. 또는 `GET /api/album` 신설 — Firestore에서 photoUrls만 모아 반환(페이지네이션 가능) |
| UI | 탭에 **「앨범」** 추가 → 그리드(날짜별 그룹 또는 전부). 클릭 시 크게 보기(라이트박스) 또는 해당 일기로 이동 |
| 성능 | 무한 스크롤 또는 월별 로딩 권장. |

**진행 순서**: 클라이언트만으로 구현(기존 diaries의 photoUrls 수집) → 전용 앨범 탭/페이지 → 그룹(연/월) 선택 시 해당 일기만 로드하도록 개선(선택).

---

## 4단계: 월별/연도별 요약 리포트 (6번) ✅ 완료

**목표**: 한 달/한 해를 한 장의 요약 리포트로 제공.

| 구분 | 내용 |
|------|------|
| API | `GET /api/report?period=month&year=2026&month=1` 또는 `period=year&year=2026`. 기존 summary API 확장 또는 전용 라우트. 기간 내 일기 수, 키워드 통계, 대표 문장 등 집계 후 LLM으로 요약 문단 생성 |
| UI | 통계 탭 또는 별도 **「리포트」** 메뉴 → 기간 선택(월/연) → 요약 카드 + 간단 차트(선택) → PDF/이미지 내보내기(선택) |
| 저장 | 리포트 생성 결과를 캐시하거나 Firestore에 리포트 문서로 저장해 재생성 부담 감소. |

**진행 순서**: report API(집계 + LLM 요약) → 리포트 전용 페이지/모달 → 기간 선택 UI.

---

## 5단계: 장소별 방문 히스토리 맵 (7번) ✅ 완료

**목표**: 방문한 장소를 지도에 표시.

| 구분 | 내용 |
|------|------|
| 데이터 | 일기에는 `location`(텍스트)만 있음. 위·경도는 사진 EXIF(photo 분석 시 저장)에만 있을 수 있음. **옵션 A**: location 문자열로 지오코딩해 좌표 변환 후 지도 표시. **옵션 B**: EXIF가 저장된 경우 해당 필드 활용 |
| API | `GET /api/map` — 일기/사진에서 location 또는 좌표 목록 반환. 또는 기존 stats API에 `locations` 좌표 리스트 추가 |
| UI | 탭에 **「지도」** 추가 → Kakao/Naver/Leaflet 등 지도 라이브러리로 마커 표시. 마커 클릭 시 해당 일기 목록 또는 요약 표시 |

**진행 순서**: 일기·사진 데이터에 좌표 존재 여부 확인 → 지오코딩 또는 EXIF 기반 좌표 수집 API → 지도 페이지 + 마커.

---

## 6단계: 가족 구성원별 성장 타임라인 (10번) ✅ 완료

**목표**: 구성원별로 일기를 타임라인 형태로 보기.

| 구분 | 내용 |
|------|------|
| API | 기존 `GET /api/timeline`에 `?member=엄마` 쿼리 추가. 또는 클라이언트에서 timeline + members 필터 |
| UI | 대시보드 탭에서 구성원 선택 시 **타임라인 뷰** 강화. 또는 별도 **「타임라인」** 탭에서 구성원 필터 + 날짜순 카드 리스트(피드와 유사하나 멤버 필터 고정) |
| 표현 | 월/년 단위 접기, “OO님의 2025년” 같은 헤더. |

**진행 순서**: timeline API에 member 필터 추가(또는 클라이언트 필터) → 타임라인 탭에서 구성원 선택 + 타임라인 카드 리스트 UI.

## 요약 — 권장 진행 순서

| 순서 | 항목 | 이유 |
|------|------|------|
| 1 | 일기 수정/편집 | 다른 기능의 기반(수정이 있으면 리포트·맵 등도 일관됨), API 한 개로 완결 |
| 2 | 오늘과 비슷한 날 | 읽기 화면 만족도 향상, API + 작은 UI 블록으로 완결 |
| 3 | 사진 앨범 뷰 | 기존 데이터만으로 구현 가능, 새 탭 하나로 완결 |
| 4 | 월별/연도별 리포트 | LLM·집계 필요하나 요청 빈도는 낮음, 별도 리포트 페이지로 분리 가능 |
| 5 | 장소 맵 | 지오코딩/EXIF 정책 결정 필요, 지도 라이브러리 도입 |
| 6 | 구성원별 성장 타임라인 | 기존 대시보드·타임라인 확장, 필터 + UI 강화 |
